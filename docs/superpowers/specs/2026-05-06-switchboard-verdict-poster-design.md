## Problem Statement

A modern tech worker juggles 8-10 apps daily — Asana, Jira, Confluence, Outlook, Slack, GitHub, Tableau, etc. The data is there but no single surface answers the only question that actually matters Monday morning: *am I making progress on what I said mattered?* Existing dashboards (Notion, Coda, Grafana) show data; nothing renders a **verdict**.

Switchboard is the verdict layer: a single page powered by MCP servers that reads from your existing work tools and renders factual progress against goals you defined. Scoreboard, not coach.

## What Makes This Cool

The product is a **screenshot-worthy weekly verdict poster**, not a dashboard. The "whoa" is the receipts-backed self-review:
- **North Star** — the one goal you committed to this week
- **Verdict** — a single factual sentence: "On track: 4/5 PRs shipped, 1 awaiting review"
- **Receipts** — actual artifacts proving the verdict (the 4 PRs, links and titles)
- **Drag** — what's slowing you down: "2 PRs waiting >36h for review"
- **Momentum** — 4-week sparkline of shipped work
- **Monday Move** — one deterministic next action

Knowledge workers already screenshot dashboards that make them look in control. Switchboard's bet: a verdict-with-receipts is the most screenshot-worthy version of that.

The deeper bet: **MCPs make this tractable for one person**. We don't write integrations. We read from the MCP servers that already exist (GitHub today, Jira/Asana/Slack/Confluence as their MCPs mature). The data plane is somebody else's problem; the judgment plane is the product.

## Constraints

- One person, one weekend for v1.
- Self-described narrow product experience — architecture must stay simple, quality must come from focus not surface area.
- v1 must be locally usable Monday morning without setup ceremony.
- v1 ships with **trustable defaults** (local-first, read-only, no secret persistence, least-privilege tokens) — not full enterprise compliance. Per revised P4, full enterprise controls (SSO, audit logs) are deferred to v1.5+ once v1 has earned daily use.

## Premises

**P1.** Verdicts are **factual statements about progress against user-defined goals** — not opinions about the person. "You shipped 4 of your 5 PR goal" is in. "You're slacking" is out. Scoreboard, not coach.

**P2.** **The data plane is decoupled from the judgment plane.** Long-term: data flows through MCP servers we don't write. *v1 exception:* GitHub data is fetched via Octokit directly behind the same `MCPAdapter` interface that v1.1 will swap to a real GitHub MCP. The architectural decoupling holds; the wire transport is an internal swap. See "Data layer decision" for rationale.

**P3.** **First user = the founder, against GitHub MCP, this weekend.** Enterprise/Darshan validation comes later. We resist the temptation to build for the employer first.

**P4 (revised after Codex pushback).** v1 ships with **trustable defaults, not enterprise ceremony**: local-first, read-only, no secret persistence, least-privilege tokens, auditable code. SSO/audit logs/compliance work begins in v1.5 once the founder has used v1 daily for two consecutive weeks.

**P5.** **No AI in v1.** Verdict logic = goals + deterministic rules. AI as a v2 power-up (the "chat to reshape your dashboard" idea).

**P6.** **Open-source self-hosted distribution.** MIT-style license, runs on `localhost`. SaaS only after usage proves the wedge.

**P7.** **Admin/team rollup is candidate v2 paid tier.** v1 must respect individual privacy and not require migration when team mode lands. No surveillance scaffolding in v1.

## Cross-Model Perspective

Codex (independent cold reader) agreed on direction (verdict layer, GitHub-first wedge, MCP-as-data-plane) and sharpened the framing in three ways:

1. **"Verdict poster" not "verdict layer"** — the product is a screenshot artifact, not a page. Concrete elements: North Star, Verdict, Receipts, Drag, Momentum, Monday Move.
2. **Re-read the user's "composable canvas" pick** — it wasn't wrong, it revealed they want *control over the shape of work*, not analytics. v1 constrains layout; v2 unlocks user-controlled layout. The emotional vibe should still feel block-y / arrangeable.
3. **Real OSS reference points worth studying** — Operately (operately.com) for the goals model ("Operately is where goals live; Switchboard is where reality argues with the goals"), github/issue-metrics for GitHub computation patterns, Grafana/Metabase for self-hosted dashboard patterns.

Codex challenged P4 (enterprise-quality from day one). Argument: premature for v1 — the right bar is local-first, read-only, no secret persistence, least-privilege tokens. **The user accepted this pushback.** P4 was revised accordingly.

## Approaches Considered

### Approach A: Verdict Poster v1 (CHOSEN)

Codex's plan, minimal viable. Single Next.js page, JSON config, one goal kind (`github_prs_merged`), real GitHub MCP adapter + mock adapter for offline UI dev. `GITHUB_TOKEN` env var. Renders one verdict + receipts + Drag + Momentum + Monday Move.

**Effort:** S (1 weekend) | **Risk:** Low

Pros: ships fast; validates the verdict-poster hypothesis with founder's own usage; local-first/read-only by construction.
Cons: only one goal kind (`github_prs_merged`); the `MCPAdapter` interface exists but only one implementation lands in v1 (Octokit-backed — see "Data layer decision" below); single-route data fetch means no caching layer beyond per-process memory.

### Approach B: Engine-first verdict platform

Same UI surface as A, but with a `VerdictEngine` core (pure functions: `GoalConfig + Receipts → Verdict`) decoupled from `MCPAdapter<T>` data adapters and React rendering. Goal-kind registry, YAML config, engine unit-tested.

**Effort:** M (2 weekends) | **Risk:** Med

Pros: matches P2 architecturally; second goal kind is a 100-line PR; testable.
Cons: one extra weekend before daily use; risk of over-engineering before validating the framing.

### Approach C: Switchboard Daily (CLI / no UI)

A CLI/daemon that runs on a schedule, queries MCPs, renders verdicts as plain markdown to stdout, with optional email/Slack delivery. The verdict is the product.

**Effort:** XS (half a weekend) | **Risk:** Very Low

Pros: maximally aligned with P1+P2; easy to actually run daily via cron.
Cons: forfeits the screenshot moment; harder to demo/evangelize; needs rewrite for v2 canvas.

## Recommended Approach

**Approach A.** Codex's plan is concrete and validated by a fresh outside read. Founder has narrow product experience by their own admission, so the highest-value move is *use it yourself for two weeks before committing to architecture*. If A's framing feels right after two Mondays, refactoring to B is naturally motivated by adding a second goal kind. A is the right *first step*; B is the right *destination*.

## v1 Implementation Sketch

### Stack
- Next.js 14 LTS (App Router) + TypeScript + Tailwind 3 *(deliberately Next 14 LTS, not 15 — Next 15 + shadcn templates lag. Tailwind 3 not 4 for the same reason.)*
- **No shadcn/ui in v1.** Hand-roll the 6 cards (≈80 lines each). Skipping shadcn init saves 1-2 hours of weekend time fighting tailwind preset + radix primitives. Add shadcn in v1.5 when there's appetite for design polish.
- **Typed TS config** (no DB in v1). The user edits `switchboard.config.ts` directly. Graduate to SQLite/Drizzle only if multi-goal lands.
- **Data layer:** see "Data layer decision" below — v1 uses `@octokit/rest` directly behind an `MCPAdapter` interface. v1.1 swaps to a real GitHub MCP server transparently.
- `GITHUB_TOKEN` env var (least-privilege: `public_repo` + `read:user` for personal use; `repo:read` if scanning private repos)
- Read-only by design; no mutations to GitHub
- Default time zone: system local TZ. Configurable via `process.env.TZ` if needed.

### Data layer decision (resolves reviewer's critical feasibility flag)

**Tension:** P2 says "MCPs are the data plane." But MCP-from-a-Next.js-API-route has a real lifecycle problem — stdio MCP servers expect a long-lived host process; Next.js route handlers are short-lived. Spawning a child per request is slow; managing a singleton across hot-reloads is fragile. This was the single biggest weekend risk.

**Resolution:** v1 honors P2's *spirit* (judgment plane decoupled from data plane) while sidestepping P2's *transport* for one release.

- Define an `MCPAdapter<T>` interface in `lib/mcp/adapter.ts` shaped to match what an MCP tool would return.
- v1 ships with a single concrete implementation: `OctokitGitHubAdapter` that talks to GitHub's REST API directly via `@octokit/rest`. No MCP transport in v1.
- v1.1 (after two weeks of daily use) adds `GitHubMCPAdapter` implementing the same interface, gated by env flag. Swapping is a one-line change in `app/api/verdict/route.ts`. The verdict engine, UI, and config don't change.
- This keeps the weekend feasible AND keeps Switchboard adapter-pattern-clean for the moment a stable corporate MCP (Darshan) shows up at v2.

This is a deliberate compromise on P2's letter to preserve P3 (ship this weekend) and the user's quality bar (don't ship a flaky MCP plumbing layer).

### Data shapes (lib/verdicts/types.ts)

```ts
export type GoalKind = "github_prs_merged"; // v2: jira_tickets_closed | slack_dms_unread | ...

export interface GoalConfig {
  kind: GoalKind;
  label: string;            // free-form prose, no semantic filtering
  target: number;           // e.g. 5
  unit: string;             // SINGULAR form: e.g. "PR". Engine pluralizes when N != 1.
  repos: string[];          // ["owner/repo", ...]
  author: string;           // GitHub login
  timeWindow?: "this_week"; // default Monday-Sunday in system TZ
                            // future: { start: string; end: string } for explicit ISO range
}

export interface Receipt {
  id: string;               // GitHub PR node_id (GraphQL-stable string, not the numeric REST id)
  title: string;
  url: string;
  repo: string;             // "owner/repo"
  mergedAt?: string;        // ISO 8601, only for merged PRs
  openedAt: string;         // ISO 8601
  hoursSinceUpdate?: number; // for open PRs: hours since GitHub's `updated_at` (proxy for any activity). v1 limitation: this includes label changes, etc., not just code/review activity. Fine for v1's purpose; v1.1 with the MCP adapter can refine.
}

// Status thresholds: shipped (≥1.0), on_track (≥0.8), nearly_there (≥0.5), behind (<0.5)
export type VerdictStatus = "shipped" | "on_track" | "nearly_there" | "behind";

export interface Verdict {
  goal: GoalConfig;
  status: VerdictStatus;
  headline: string;         // deterministic template (see "Verdict status bands")
  actual: number;
  target: number;
  receipts: Receipt[];      // the merged PRs that count toward target
  drag: Receipt[];          // open PRs matching the Drag rule
  momentum: number[];       // 4 weekly counts, oldest first; index 3 = current (partial) week
  mondayMove: string | null; // pre-formatted sentence; null if no clear move
}

export interface SwitchboardConfig {
  goals: GoalConfig[];      // v1: exactly one
}
```

### Adapter interface (lib/mcp/adapter.ts)

```ts
export type AdapterErrorCode =
  | "auth_failed"
  | "rate_limited"
  | "network"
  | "not_found"      // repo/user doesn't exist or token can't see it
  | "unknown";

export interface AdapterError {
  code: AdapterErrorCode;
  message: string;
  retryAfterSeconds?: number; // for rate_limited
}

// Discriminated union — adapter functions never throw on expected failures.
export type AdapterResult<T> =
  | { ok: true; data: T }
  | { ok: false; error: AdapterError };

export interface MCPAdapter {
  listMergedPRs(args: {
    repos: string[];
    author: string;
    since: string;          // ISO 8601
    until: string;
  }): Promise<AdapterResult<Receipt[]>>;

  listOpenPRs(args: {
    repos: string[];
    author: string;
  }): Promise<AdapterResult<Receipt[]>>;
}
```

`OctokitGitHubAdapter` (v1) and `GitHubMCPAdapter` (v1.1) both implement this. `MockAdapter` returns fixture data. **Multi-repo strategy:** the v1 Octokit implementation uses `octokit.rest.search.issuesAndPullRequests` with a single query (`is:pr author:X merged:>=since repo:R1 repo:R2 ...`) to avoid N-per-repo fan-out. Hard cap: 100 results. If exceeded, return what we have plus a flag in the response.

**`hoursSinceUpdate` computation (v1 simplification):** the adapter populates this from GitHub's `updated_at` field returned by the search API directly. No per-PR fan-out — we don't fetch reviews or commits separately. **Limitation acknowledged:** `updated_at` ticks on any change including labels and assignment, so it's a noisy proxy for "review activity." Acceptable for v1 because:
- a noisy proxy still surfaces stale PRs correctly most of the time;
- the alternative (per-PR `pulls.listReviews` + `pulls.listCommits`) would explode the rate-limit budget and add 1-2 hours of weekend coding;
- v1.1's real GitHub MCP adapter can compute the precise version because the MCP server has more efficient access patterns.

**"Stale" for Drag:** open PRs where `hoursSinceUpdate > DRAG_THRESHOLD_HOURS`. This catches both "no review yet" and "approved-then-author-stalled" via the same simple rule. Merged PRs are never Drag.

**"Shipped" definition:** merged-to-default-branch only. The Octokit search filter `is:merged` covers this; closed-without-merge does not count.

### Verdict status bands (deterministic, no AI)

```ts
function statusFor(actual: number, target: number): VerdictStatus {
  const ratio = actual / target;
  if (ratio >= 1.0) return "shipped";       // hit or exceeded the goal
  if (ratio >= 0.8) return "on_track";      // 80%+ of target
  if (ratio >= 0.5) return "nearly_there";  // halfway or better
  return "behind";                          // less than halfway
}
```

Headline templates (`{unit}` and `{actual}`/`{target}` substitute from `GoalConfig`):
- `shipped`: "Shipped: {actual}/{target} {unit} this week."
- `on_track`: "On track: {actual}/{target} {unit} this week."
- `nearly_there`: "Halfway: {actual}/{target} {unit} this week." *(renamed from "Nearly there" — at 50% the band copy "Nearly there" was misleading; "Halfway" is honest)*
- `behind`: "Behind: {actual}/{target} {unit} this week."

If `drag.length > 0`, append a single sentence: ` {N} {unit} {pluralize(N, "is", "are")} stale (waiting >{DRAG_THRESHOLD_HOURS}h).` where `pluralize(n, sing, plur)` is a simple helper in `lib/verdicts/engine.ts`.

**Worked example (4/5 PRs, 1 stale open PR):** ratio = 0.8 → `on_track` → headline = `"On track: 4/5 PRs this week. 1 PR is stale (waiting >24h)."`

### Drag rule (v1)

`Drag` = open PRs authored by `author` in configured `repos` where `hoursSinceUpdate > DRAG_THRESHOLD_HOURS`. v1 default: `DRAG_THRESHOLD_HOURS = 24`. (v1.1 with the real MCP adapter will refine this to use precise review/push timestamps.)

### Monday Move rule (v1)

`pickMondayMove(drag, openPRs): string | null` — implemented in `lib/verdicts/engine.ts`. The engine returns the pre-formatted sentence; the component just renders it. Logic:

1. If `drag.length > 0`: pick the item with greatest `hoursSinceUpdate`. Format: `"Unblock {repo}#{prNumber} — stale {N}h."`
2. Else if `openPRs.length > 0`: pick the most recently opened. Format: `"Push {repo}#{prNumber} — {N}h since you opened it."`
3. Else: `null` (component renders nothing).

### Momentum (resolves reviewer's "no DB vs 4-week sparkline" flag)

Computed live each request via a single `listMergedPRs` call with `since = now - 28 days`, then bucketed into 4 ISO weeks (Monday 00:00 — Sunday 23:59 in system TZ). The 4 buckets are: 3 trailing completed weeks + 1 current (partial) week as bucket index 3.

No persistence in v1. Cached for 1 hour in a module-level `Map` keyed by `${author}::${repos.sort().join(",")}::${weekStartIso}` (week-start in the key prevents day-boundary collisions). Cache TTL is intentional and orthogonal to the API's `Cache-Control: no-store` (which prevents browser/CDN caching of personalized data).

**Dev-mode behavior:** in `process.env.NODE_ENV !== "production"`, the cache is disabled — Next.js hot-reload + module-level mutable singletons is a known footgun and the dev experience improves by re-fetching every request. In `next start` (production / `localhost` deploy) the cache is on; v1.5 adds LRU eviction if memory becomes a concern.

**Sparkline rendering:** `MomentumSparkline.tsx` is a hand-rolled inline `<svg>` (≈40 lines) — no chart library dependency. Just `<polyline>` over normalized counts, with the current week dot rendered slightly transparent to indicate it's partial.

### API contract (app/api/verdict/route.ts)

- `GET /api/verdict` — no params (config is read from `switchboard.config.ts`).
- Response: `Verdict` (above), `Cache-Control: no-store`.
- On adapter error: `{ error: "string", code: "auth_failed" | "rate_limited" | "network" | "unknown" }`, status 502.
- On empty target (target=0 or no goal configured): status 400 with friendly message.

### Failure-state UX

- **Auth failed** (token missing/invalid): page renders "Switchboard couldn't reach GitHub. Check that `GITHUB_TOKEN` is set and has `read:user` + `public_repo` scopes." plus a one-line setup link.
- **Rate limited**: "GitHub rate limit hit. Try again in {minutes}m." Show the GitHub-provided reset time.
- **Network error**: "Can't reach GitHub right now." Plus retry button.
- **Empty result** (zero merged PRs in window): render the verdict normally — `"Behind: 0/5 PRs this week."` is the correct verdict at zero. The whole point of a scoreboard is showing zero. Drag and Monday Move still render if applicable.

### MockAdapter contract

`MockAdapter` reads a fixed JSON fixture (`lib/mcp/fixtures.ts` exports a `MOCK_PRS` array with realistic `Receipt` shape) and **does** filter by the args: `listMergedPRs` returns only fixtures with `mergedAt` between `since` and `until`; `listOpenPRs` returns only fixtures with no `mergedAt`. This makes the mock useful for dev UI work because the verdict respects the window like the real adapter does.

## Non-Goals (v1)

- **No historical week archive.** v1 always shows "current week." If today is Wednesday, you see this Monday-Sunday's progress so far; you cannot scroll back to see last week's verdict. v1.5 may add archival.
- **No multi-goal config.** `goals` array is exactly one entry. Multi-goal is v1.1 once the engine is generalized.
- **No notifications.** No email, no Slack DM, no push. The page exists; you open it. (The "Switchboard Daily" CLI form is parking-lot if email-on-Monday becomes important.)
- **No team or multi-user mode.** Strictly single-user, single-machine.

### Security model (sketch for the README)

- Token in `GITHUB_TOKEN` env var only. Never written to disk, never logged, never sent off-host.
- No telemetry. No analytics. No remote calls except to `api.github.com` via Octokit.
- All data is computed and rendered in-process. No persistence in v1.
- Server runs on `localhost`, no public binding by default.
- All operations are read-only.

No human/team dependencies — single-person project.
